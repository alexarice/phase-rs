12* [numpy `__array_interface__` documentation](https://numpy.org/doc/stable/reference/arrays.interface.html#python-side)

14## Simple primitive types
15
16Integers and floats correspond to simple dtypes:
17
18### Integers
19
20* The rust types `i8`, `i16`, `i32`, `i64` use type code `i`.
21* The rust types `u8`, `u16`, `u32`, `u64` use type code `u`.
22
23**Notice:** numpy does not support 128-bit integers</li>
24
25### Floats
26
27* The rust types `f32`, `f64` use type code `f`.
28* When the **`"half"`** feature is enabled, [`f16`] is also supported.
29
30**Notice:** numpy *does* have 128-bit floats, but it is not currently supported by `npyz`.
31
32### Complex
33
34When the **`"complex"`** feature is enabled, rust types [`Complex32`] and [`Complex64`] may use type code `c`.
35
36**Notice:** numpy does have have complex numbers backed by 128-bit floats, but this is not supported by `npyz`.
37
38### Bool
39
40The rust type `bool` may be serialized as `|b1`.
41
42### Endianness
43
44In all of the above cases, npyz uses the machine endianness by default when serializing, but
45supports serializing and deserializing as any endianness.  The size of the datatype in the file
46must match the size of the rust type used.
47
48## Date and time
49
50There are two type codes used by numpy for time and date.
51
52* `m`: A `numpy.timedelta64`.
53* `M`: A `numpy.datetime64`.
54
55Both of these are represented as 8-byte signed integers, and therefore can use **`i64`** in rust.
56
57The type strings for these types must specify units in square brackets, e.g. `"<m8[ns]"` for
58nanoseconds.
59
60## String and blob types
61
62### Overview
63
64There are three type codes for variable-sized strings of data found in npy files:
65
66* `|VN`: A fixed-size array of `N` bytes.
67* `|SN` (or `|aN`): A possibly-null-terminated sequence of bytes of length `<= N`.
68* `<UN`: An array of `N` Unicode code points, each encoded as a 4-byte integer.
69
70Each will be described in its own section further below.
71The following support matrix shows how various rust types may serialize as these type codes.
72
73| rust type               | feature      | `VM` | `SM`/`aM` | `UM` |  [`AutoSerialize`] dtype | notes |
74|:---------               | ------------ | --- | ------- | --- | ------------------ | :--- |
75| `String`/`str`          |              | ❌ | ✅ | ✅ | ➖   | |
76| `Vec<u8>`/`[u8]`        |              | ✅ | ✅ | ❌ | ➖   | length must `== M` when writing `V` |
77| `Vec<u32>`/`[u32]`      |              | ❌ | ❌ | ✅ | ➖   | most general type to read `U` |
78| `Vec<char>`/`[char]`    |              | ❌ | ❌ | ✅ | ➖   | |
79| [`FixedSizeBytes`]`<N>` |              | ✅ | ❌ | ❌ | `VN` | requires `N == M` |
80| [`ArrayVec`]`<u8, N>`   | `"arrayvec"` | ✅ | ✅ | ❌ | ➖   | `VM` requires `M <= N` upfront <br/> `S`/`a` truncates when reading |
81| [`ArrayVec`]`<u32, N>`  | `"arrayvec"` | ❌ | ❌ | ✅ | `UN` | truncates when reading |
82| [`ArrayVec`]`<char, N>` | `"arrayvec"` | ❌ | ❌ | ✅ | `UN` | truncates when reading |
83| [`ArrayString`]`<N>`    | `"arrayvec"` | ❌ | ✅ | W  | `SN` | truncates when reading |
84
85Legend:
86* ❌: Cannot read or write as this dtype.
87* ✅: Can read and write as this dtype.
88* ➖: This type does not implement [`AutoSerialize`].
89* W: Can be written but not read.
90
91### Raw byte blobs (`|VN`)
92
93This is the simplest sequence type.  It is a blob of exactly `N` bytes.
94
95This type is most easily read as `Vec<u8>`.  However, if `N` is known at compile-time, then
96the individual heap allocations per item can be avoided by using [`FixedSizeBytes`] instead;
97this is a newtype wrapper around `[u8; N]`.
98
99(you cannot use `[u8; N]` directly because this would be ambiguous in a structured array;
100 see the section on "Array members")
101
102### Unicode strings (`<UN`, `>UN`)
103
104This is the type natively used by numpy for Python 3's `str`.
105
106It is an array of `N` Unicode code points, each encoded as a 4-byte integer.
107Trailing null code points are not considered to be part of the content. (but interior nulls are)
108
109Notice these are "code points" and not "scalar values"!
110One could think of this format as "UTF-32, but surrogates are allowed".
111The following Rust types are supported:
112
113* `Vec<u32>`, which is able to read any valid `U` value from a file.
114* `Vec<char>`, which will fail on reading surrogates.
115* `String`, which will fail on reading surrogates.
116
117Notice that `String` also alternatively supports `|SN` if you want a more compressed representation
118in the file, however this is a non-standard convention (see the section on `|SN` for more details).
119
120### Possibly-null-terminated byte blobs (`|SN`)
121
122This is a legacy string type that roughly corresponds to Python 2's `str`.
123
124This is similar to `|VN`, but trailing 0 bytes are not considered to be part of the content.
125(Interior null bytes, on the other hand, are part of the content).  numpy itself places no
126restrictions on the contents of the bytes.
127
128The most natural type to decode this with is `Vec<u8>`.
129
130However, npyz also supports `String` for this type.  When reading `S` into `String`, the contents
131must be valid UTF-8, or reading will fail.  Using `S` for UTF-8 encoded strings is not
132a standard practice in python, so beware when using this on files from unknown sources!
133
134### `"arrayvec"` feature
135
136The types described above for `S` and `U` all require individual heap allocations for each item,
137which may be costly in terms of memory.  To address this, one may enable the `"arrayvec"` feature
138to allow the serialization/deserialization of types from [`arrayvec`].
139
140These types have the additional benefit that they implement [`AutoSerialize`] (by defaulting to the
141const parameter `N` as their size), though they all support dtypes with arbitrary size.
142
143When reading a string that is too long to fit in the destination type, it will be truncated.
144The justification is that there is no straightforward way for downstream code to implement such
145recovery behavior on their own.
146Truncation is only performed on `S`, `a`, and `U` data (never on `V` data), and always produces a
147valid value of that type.  (e.g. `ArrayString<N>` will truncate to a valid UTF-8 prefix, while
148`ArrayVec<u8, N>` will truncate to arbitrary bytes).
149
150If you do not wish to allow truncation to occur, you may check that [`TypeStr::size_field`]`() <= N`.
151(`N` being the const length of the arrayvec)
152
153The [`arrayvec`] types support almost all of the same operations as their heap-allocated
154counterparts. However, reading `U` into [`ArrayString`] is expressly forbidden, because
155due to the transcoding, there is no reliable way for the caller to detect when truncation may
156have occurred.
157
158## Structured arrays
159

161macros for [`Serialize`], [`Deserialize`], and [`AutoSerialize`].

170}
